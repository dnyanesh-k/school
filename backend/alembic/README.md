VidyaTrack — Alembic migration guide
=====================================

Alembic tracks every schema change so you can apply, roll back, and audit
database migrations safely across local dev, staging, and Supabase.

─── Quick reference ──────────────────────────────────────────────────────────

All commands run from the backend/ folder with the venv active.

  Generate migration after a model change:
    alembic revision --autogenerate -m "short description of change"

  Apply all pending migrations:
    alembic upgrade head

  Roll back one migration:
    alembic downgrade -1

  Roll back to a specific revision:
    alembic downgrade <revision_id>

  Show current DB revision:
    alembic current

  Show full migration history:
    alembic history --verbose

─── How it works ─────────────────────────────────────────────────────────────

Alembic stores the current schema version in the alembic_version table
(lives in the public schema of school_db). You can verify it with:

  SELECT * FROM alembic_version;

If the table is missing in pgAdmin, do:
  1. Right-click the database → Refresh
  2. Look under Schemas → public → Tables → alembic_version
  3. Or run the SELECT above in pgAdmin Query Tool

─── Local vs Supabase ────────────────────────────────────────────────────────

Switch DATABASE_URL in backend/.env (comment/uncomment):

  # Local (active)
  DATABASE_URL=postgresql+asyncpg://postgres:root@localhost:5432/school_db

  # Supabase (uncomment when deploying)
  # DATABASE_URL=postgresql+asyncpg://postgres.[ref]:[pass]@supabase...

For a fresh Supabase DB:
  1. Start the server once — create_all builds all tables
  2. Server auto-stamps alembic_version at head
  3. Future deploys: run "alembic upgrade head" before starting

─── File layout ──────────────────────────────────────────────────────────────

  alembic/
    env.py            — async SQLAlchemy config, reads .env via app.core.config
    script.py.mako    — template for generated migration files (do not edit)
    versions/         — auto-generated migration files (one per schema change)
  alembic.ini         — Alembic configuration (DB URL injected from env.py)

─── Rules ────────────────────────────────────────────────────────────────────

  - Never edit a migration file that has already been applied to any shared DB.
  - Always review autogenerated files before running upgrade head.
  - Autogenerate may miss: custom SQL defaults, check constraints, partial indexes.
    Add these manually to the generated file if needed.
  - Commit migration files to git — they are the source of truth for schema.
